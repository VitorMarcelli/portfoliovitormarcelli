# PRD — Novo Portfólio de Vitor Marcelli

## 1. Visão Geral do Produto

### 1.1 Nome do produto
Novo Portfólio de Vitor Marcelli

### 1.2 Tipo de produto
Aplicação web responsiva de apresentação profissional, posicionamento técnico e vitrine de projetos.

### 1.3 Objetivo principal
Construir um portfólio digital de alto impacto visual, com identidade autoral forte, experiência premium e percepção imediata de autoridade técnica.

### 1.4 Objetivo estratégico
Transformar o portfólio em uma ferramenta ativa de posicionamento profissional, geração de oportunidades e validação de competência técnica, não apenas em uma página institucional.

### 1.5 Contexto
O usuário deseja um novo portfólio para se apresentar a empresas, exibir projetos, sua apresentação profissional e os serviços que executa. A principal intenção do produto é superar portfólios comuns e entregar uma experiência memorável, sofisticada e profissional.

Além da camada visual, existe a necessidade de facilitar a manutenção da seção de projetos, idealmente com atualização simplificada e preparada para integração futura com fontes externas.

---

## 2. Problema que o Produto Resolve

Hoje, um portfólio convencional frequentemente falha em três pontos:

1. Não gera impacto memorável nos primeiros segundos.
2. Não posiciona o profissional como alguém acima da média.
3. Dá trabalho para manter atualizado, especialmente na seção de projetos.

O novo portfólio deve resolver isso entregando:
- presença visual forte;
- narrativa profissional clara;
- organização de projetos com percepção premium;
- manutenção simples;
- arquitetura pronta para crescer.

---

## 3. Objetivos do Produto

### 3.1 Objetivos primários
- Gerar forte impacto visual nos primeiros segundos de navegação.
- Posicionar Vitor Marcelli como desenvolvedor fullstack com perfil criativo, técnico e orientado a solução.

- Preparar integração futura de projetos com GitHub.
- Possibilitar futuramente um painel ou fluxo de cadastro de projetos.
- Reforçar branding pessoal.
- Servir como ativo comercial e institucional.

### 3.3 Resultado esperado
Quando alguém acessar o site, deve perceber imediatamente:
- alto nível de capricho;
- domínio técnico;
- bom gosto visual;
- clareza sobre o que o profissional faz;
- confiança para entrar em contato.

## 7. Escopo Funcional

### 7.1 Estrutura macro do produto
O sistema deverá conter, no MVP:

1. Página inicial com hero de impacto.
2. Seção de apresentação pessoal.
3. Seção “o que eu faço”.
4. Seção de projetos.
5. Seção de stack / tecnologias.
6. Seção de diferenciais.
7. Seção de contato.
8. Navegação fluida entre seções.
9. Responsividade completa.
10. Base preparada para integração futura de projetos com fontes externas.

---

## 8. Requisitos Funcionais

### RF-01 — Hero principal de alto impacto
O sistema deve exibir uma seção inicial visualmente marcante com:
- nome do profissional;
- headline de posicionamento;
- texto curto de apresentação;
- CTA para projetos;
- CTA para contato;
- elementos visuais e/ou animações que reforcem a identidade premium.

### RF-02 — Apresentação profissional
O sistema deve apresentar uma seção “Sobre mim” com:
- resumo profissional;
- áreas de atuação;
- diferenciais de trabalho;
- posicionamento claro como desenvolvedor fullstack e construtor de soluções.

### RF-03 — Seção “O que eu faço”
O sistema deve exibir categorias claras de atuação, como:
- desenvolvimento web;
- desenvolvimento de sistemas sob medida;
- automações;
- integrações;
- APIs;
- inteligência artificial;
- sites institucionais e páginas de alta conversão;
- aplicativos e soluções específicas.

### RF-04 — Seção de projetos
O sistema deve possuir uma vitrine de projetos com:
- cards ou blocos de destaque;
- nome do projeto;
- descrição curta;
- tecnologias utilizadas;
- categoria;
- link para repositório, demonstração ou detalhes;
- priorização visual de projetos mais relevantes.

### RF-05 — Filtro ou organização de projetos
O sistema deve permitir organização por categorias, destaque ou lógica equivalente, com foco em facilitar leitura e exploração.

### RF-06 — Curadoria de projetos
O sistema não deve obrigatoriamente exibir todos os projetos disponíveis. Deve existir uma camada de curadoria para selecionar apenas os projetos estratégicos.

### RF-07 — Metadados complementares por projeto
O sistema deve permitir complementar o projeto com dados como:
- título amigável;
- descrição comercial;
- imagem de capa;
- categoria;
- status;
- destaque;
- link de demonstração.

### RF-08 — Seção de stack
O sistema deve exibir tecnologias e ferramentas dominadas, agrupadas por contexto, como:
- frontend;
- backend;
- mobile;
- banco de dados;
- cloud/devops;
- automações/IA.

### RF-09 — Seção de diferenciais
O sistema deve comunicar diferenciais competitivos, como:
- capacidade de entregar ponta a ponta;
- integração entre negócio e tecnologia;
- experiência com sistemas reais;
- foco em solução e execução.

### RF-10 — Contato
O sistema deve exibir meios de contato com alta visibilidade, incluindo:
- LinkedIn;
- GitHub;
- e-mail;
- WhatsApp, se desejado.

### RF-11 — Navegação intuitiva
O sistema deve possuir navegação clara, fluida e agradável entre seções.

### RF-12 — Responsividade
O sistema deve ter excelente comportamento em desktop, tablet e mobile.

### RF-13 — Performance visual e técnica
O sistema deve carregar de forma rápida e manter fluidez nas interações.

### RF-14 — Preparação para evolução futura
A arquitetura deve permitir, em fases futuras:
- página individual por projeto;
- painel administrativo;
- depoimentos;
- blog ou artigos;
- CMS leve;
- analytics.

---

## 9. Requisitos Não Funcionais

### RNF-01 — Identidade visual premium
O produto deve ter acabamento visual acima da média, evitando aparência genérica.

### RNF-02 — Alta percepção de qualidade
Os detalhes visuais, microinterações, tipografia e ritmo de layout devem transmitir produto profissional.

### RNF-03 — Manutenibilidade
A base deve permitir atualização simples de conteúdo e projetos.

### RNF-04 — Escalabilidade de conteúdo
Novos projetos e novas seções não devem exigir retrabalho estrutural relevante.

### RNF-05 — SEO básico
O site deve ter estrutura mínima de SEO para facilitar indexação e compartilhamento.

### RNF-06 — Acessibilidade mínima
O site deve respeitar boas práticas mínimas de contraste, legibilidade e navegação.

### RNF-07 — Deploy simples
A aplicação deve ser facilmente publicável e atualizável.

## 11. Modelo de Gestão dos Projetos

### 11.1 Abordagem recomendada
Modelo híbrido com curadoria.

### 11.2 Como deve funcionar
- O sistema poderá consumir dados de fontes externas no futuro.
- Apenas projetos selecionados entram na vitrine.
- Dados externos poderão ser enriquecidos por metadados locais.

### 11.3 Justificativa
Nem todo repositório ou projeto fortalece imagem profissional. A curadoria evita poluição visual e aumenta percepção de qualidade.

### 11.4 Formas possíveis de seleção
O sistema poderá no futuro usar uma destas lógicas:
- arquivo local de configuração;
- lista manual de projetos;
- tópicos/tags;
- projetos fixados;
- futura interface administrativa.

## 13. Fluxos Principais

### Fluxo 1 — Primeira visita
1. Usuário entra no site.
2. Recebe impacto visual imediato no hero.
3. Entende rapidamente quem é Vitor.
4. Navega para projetos.
5. Analisa trabalhos.
6. Vai para contato.

### Fluxo 2 — Recrutador avaliando perfil
1. Abre o portfólio por link enviado.
2. Lê headline e resumo.
3. Avalia stack e diferenciais.
4. Verifica projetos.
5. Decide entrar em contato.

### Fluxo 3 — Atualização de projetos
1. Novo projeto é criado ou selecionado.
2. Projeto recebe curadoria.
3. Metadados são adicionados ou ajustados.
4. O site passa a exibi-lo corretamente na seção de projetos.

## 14. Critérios de Sucesso

O produto será considerado bem-sucedido se:
- causar forte impacto visual inicial;
- transmitir profissionalismo e diferenciação;
- organizar os projetos com clareza;
- facilitar a apresentação do usuário a empresas;
- permitir crescimento sem refação estrutural;
- manter atualização de projetos de forma simples.

---

## 15. Riscos e Cuidados

### Risco 1 — Excesso de efeitos visuais
Pode prejudicar leitura, performance e clareza.

**Mitigação:** priorizar elegância e direção de arte com controle.

### Risco 2 — Visual bonito, mas sem posicionamento claro
Pode impressionar, mas não converter.

**Mitigação:** manter clareza de proposta, headline e narrativa.

### Risco 3 — Exposição de projetos irrelevantes
Pode enfraquecer a imagem profissional.

**Mitigação:** manter camada de curadoria manual ou semiautomática.

### Risco 4 — Manutenção difícil
Pode fazer o site envelhecer rápido.

**Mitigação:** estruturar conteúdo e arquitetura modular desde o início.
